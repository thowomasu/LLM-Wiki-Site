It predicts a fixed-grid EIS spectrum from earlier time-series current and voltage features, then tests that prediction on a whole dataset that was held out before scoring.

Full Handoff

Cross Dataset Pulse To EIS Training

This folder is the supervised cross-dataset validation scaffold.

Plain English: it tries to predict an EIS spectrum from time-domain current and voltage behavior measured before that EIS spectrum. EIS means Electrochemical Impedance Spectroscopy. The target is a fixed frequency grid with real impedance and negative-imaginary impedance values.

This is not the frozen CALB pulse-to-EIS rule. That older rule is the internal CALB rule that checks whether HPPC pulse windows recover EIS-like information. This folder is a separate supervised model scaffold for testing whether similar time-series-to-EIS structure survives across datasets.

Current Verdict

The scaffold is now viable for the next similar internet dataset.

Why that is true:

1. It has a unified row schema for pairing one time-domain file with one later same-cell EIS spectrum.

2. It blocks random row splits and uses whole-dataset holdouts.

3. It builds a fixed-grid EIS feature/target matrix.

4. It trains only on development datasets with already-seen outcomes.

5. It freezes a model and baseline before scoring a new dataset.

6. It scored Cambridge once without post-outcome tuning.

Current Cambridge result: partial_support.

That means the locked model beat the locked global-mean EIS baseline, but the improvement was too small for a strong external-support verdict.

Do not oversell this. It is real progress. It is not a win lap.

What The Pipeline Uses

One supervised sample means:

  • Input: current, voltage, time, temperature, SOC, and capacity-style summaries that are available before the EIS target.
  • Target: the same cell/channel EIS spectrum measured after or near that time-domain segment.
  • Leak guard: train/test splits must hold out whole cells, diagnostic rounds, or whole datasets. No random row split.

The fixed EIS target grid is:

[0.05, 0.1, 0.2, 0.5, 1.0, 2.0, 5.0, 10.0, 20.0, 40.0] Hz

For each frequency, the target includes:

  • Zreal in ohms.
  • -Zimag in ohms.

That gives 20 target columns.

How To Use This For The Next Internet Dataset

Use this checklist before downloading a huge archive:

1. Same-cell key exists.

The files must expose a cell ID, channel ID, sample ID, or another key that joins time-series data to EIS.

2. Time-series side exists.

The dataset needs current, voltage, and time. Temperature, SOC, and capacity are useful but not always required.

3. EIS side exists.

The dataset needs frequency, real impedance, and imaginary or negative-imaginary impedance.

4. The time-series data comes before or near the EIS target.

Future data must not be used to predict a past EIS spectrum.

5. The dataset can be held out as a whole dataset.

If you tune after looking at its validation metrics, it is no longer clean validation.

6. The target frequency range overlaps the fixed grid.

At minimum, enough measured EIS frequencies must cover the grid from 0.05 Hz to 40 Hz after interpolation.

7. The result can be compared against a boring baseline.

If the model cannot beat the global training mean or a simple voltage/SOC/temperature baseline, call it weak or no support.

Minimum useful download target:

  • Raw or processed battery files.
  • Same-cell EIS and time-series measurements.
  • Clear cell/channel/sample labels.
  • Temperature or protocol labels.
  • Enough rows to hold out the full dataset.

Bad target:

  • HPPC or cycling only, no EIS.
  • EIS only, no time-series current/voltage data.
  • Files with no join key.
  • A dataset that requires hand-tuning after seeing the validation result.

```powershell
& "[local path redacted]" -B scripts\experiments\cross_dataset_pulse_to_eis_training\run_cambridge_one_shot_validation.py

```

For a future dataset, do not reuse the Cambridge runner name as if the new dataset is Cambridge. Make a new frozen config and a new one-shot runner, then record the exact hashes and verdict rules before scoring.

Claim Boundary

Allowed:

  • The supervised cross-dataset scaffold is viable for another similar dataset.
  • Cambridge gives partial support to the scaffold because the locked model beat the locked mean baseline.
  • The Cambridge improvement is weak, about 1.67% RMSE improvement, so it is not strong external support.

Not allowed:

  • The frozen CALB pulse-to-EIS rule is externally validated.
  • The Cambridge result rescues Panasonic, RADAR4KIT, or Stanford frozen-rule failures.
  • Stanford is clean validation for the selected model.
  • Post-Cambridge model changes are still clean Cambridge validation.

If you change features, target grid, model, split roles, alpha, or verdict rules after seeing Cambridge, label the next run as post-Cambridge development.
