Stakeholder technical demo

Cross-Dataset Pulse-to-EIS Demo Pipeline

This presentation shows the actual demo pipeline, the math behind the target and scoring, the numbers produced by the current run, and the claim boundary. It assumes the reader already knows why EIS, impedance spectra, and relaxation behavior matter.

Executive Read

The demo is a small but real supervised pipeline. It takes time-domain battery features, predicts a fixed-grid EIS vector, compares against a global-mean baseline, and runs a Kramer-Kronig quality screen on the EIS target rows.

Blunt boundary: this is a working demo and stakeholder explanation, not a new external-validation claim. VM connection is deferred until the colleague-provided VM exists.

Objective, Example, Goal, Conclusion

Objective

Prove the demo code path can convert earlier time-domain behavior into a fixed-grid EIS prediction and score it honestly.

Example

Train on 120 development rows, score 30 Molicell holdout rows, and inspect 20 target columns.

Goal

Give stakeholders enough technical detail to understand the data contract, leakage guard, model, baseline, math, and failure modes.

Conclusion

The local demo works and beats the baseline. The result is useful, but it should not be oversold as a new validation claim.

Methodology And Pipeline Architecture

1. Input matrix

Read development features and Molicell holdout features. Each row is a same-cell time-domain segment paired with an EIS target.

2. Feature layer

Use current, voltage, time, temperature, SOC, and derived pulse statistics available before the EIS target.

3. Target layer

Predict real impedance and negative-imaginary impedance on a fixed frequency grid.

4. Scoring layer

Compare global-mean baseline against a KNN shrinkage model using RMSE across all target columns.

5. Physics screen

Run finite-range RC-basis Kramer-Kronig consistency on the holdout EIS targets.

Feature-To-Target Math

The supervised learning problem is vector-valued. For each sample i, we map time-domain features x_i to a fixed-grid EIS vector y_i.

x_i = features(current, voltage, time, temperature, SOC, capacity summaries)
y_i = [Zreal(f_1), -Zimag(f_1), ..., Zreal(f_10), -Zimag(f_10)]
f = [0.05, 0.1, 0.2, 0.5, 1, 2, 5, 10, 20, 40 Hz]

The target is not one scalar health label. It is a 20-dimensional impedance vector. That matters because a model can improve the easy real component while failing the negative-imaginary component if the scoring is too loose.

Baseline And Model

Boring baseline

global train target mean

y_hat_baseline = mean(Y_train)

This is the right first opponent. If the model cannot beat this, the pipeline has no practical signal yet.

Demo model

KNeighborsRegressor(n_neighbors=3) blended 20 percent with the KNN prediction and 80 percent with the train mean

y_hat_demo = 0.8 * y_hat_baseline + 0.2 * y_hat_KNN

The shrinkage keeps the demo conservative. It prevents a tiny nearest-neighbor fit from pretending it has broad generalization.

Scoring Math

The demo reports RMSE across the full fixed-grid EIS vector. Lower is better.

RMSE = sqrt(mean((Y_true - Y_pred)^2))
relative improvement = (RMSE_baseline - RMSE_model) / RMSE_baseline

Kramer-Kronig Quality Screen

The demo also checks whether the EIS targets look broadly compatible with a finite RC-basis impedance model over the measured frequency window. This is target-quality metadata, not a model-performance claim.

Z_KK(f) approx R_inf + sum_k R_k / (1 + j * 2*pi*f*tau_k)
residual_norm = RMSE(Z_measured, Z_KK_fit) / scale(Z_measured)
KK valid rows
30 / 30

The current demo subset has 30 valid rows out of 30. That supports target sanity for the demo subset. It does not prove the predictor is good by itself.

What This Proves, And What It Does Not

Supported

  • The local code path runs end to end.
  • The demo model beats a global-mean baseline on the bounded subset.
  • The EIS targets in the demo subset pass the KK screen.
  • The public site can host the presentation and JSON outputs.

Not supported

  • This is not a new external validation claim.
  • This does not validate the frozen CALB pulse-to-EIS rule.
  • This does not solve the RADAR4KIT target-quality problem.
  • This does not claim VM connection. VM setup is deferred.
